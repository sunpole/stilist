# SUNPOLE HUB — видение, цель и маршрут

## Главная идея

`SUNPOLE HUB` — это не просто страница со ссылками. Это визуальный, интерактивный и технически аккуратный каталог всех мини-веб-проектов в репозитории `stilist`.

Цель — получить удобную витрину проектов, где каждый проект виден как карточка с описанием, статусом, тегами, скриншотом и кнопкой запуска.

## Конечная цель

В итоге должен получиться полный рабочий комплекс:

1. Пользовательская витрина `SUNPOLE HUB`.
2. Ручная админка для добавления и редактирования карточек.
3. Локальный инструмент на ПК для автоматического анализа репозитория.
4. Генерация правильной базы проектов.
5. Генерация/обновление скриншотов проектов.
6. Экспорт готовых данных для GitHub Pages.
7. Безопасный архив старых версий без удаления файлов.
8. Единая кнопка/панель возврата из каждого проекта обратно в HUB.

## Главный принцип безопасности

Архив — это статус, а не удаление.

Файл остаётся на месте. Ссылка остаётся рабочей. Проект просто получает статус:

- `main` — основной проект;
- `experiment` — эксперимент;
- `archive` — архивная версия;
- `review` — требуется проверка.

## Что важно сохранить

Нельзя потерять:

- старые версии проектов;
- рабочие ссылки;
- скриншоты;
- ручные описания;
- теги;
- историю решений;
- возможность отката.

## Что нравится в будущем интерфейсе

Будущий HUB должен быть визуальным и интерактивным:

- большие кликабельные карточки;
- скриншоты проектов;
- цветные индикаторы статуса;
- категории;
- теги;
- облако тегов;
- фильтр по тегу при клике;
- фильтр по статусу;
- фильтр по категории;
- поиск по названию, описанию, тегам и техническим данным;
- понятные подписи;
- технические сведения для каждого проекта;
- связь основной версии с архивными версиями;
- красивые кнопки и бейджи;
- визуальные индикаторы качества, статуса и назначения;
- микроанимации без перегруза;
- дизайн с ощущением аккуратной витрины, а не случайного списка ссылок.

## Глобальная кнопка возврата в HUB

После утверждения финального вида HUB нужно добавить в каждый проект небольшую ненавязчивую панель/уведомление со ссылкой обратно в магазин/хаб.

Идея:

- в каждом проекте появляется маленькая плавающая кнопка или уведомление;
- текст примерно: `← Вернуться в SUNPOLE HUB`;
- пользователь может закрыть уведомление;
- после закрытия оно не появляется до обновления страницы;
- кнопка не должна мешать пользоваться инструментом;
- на мобильном она должна быть компактной;
- для старых проектов подключение должно быть максимально простым.

Возможная реализация:

```html
<script src="assets/hub/hub-return-widget.js"></script>
```

Файл `hub-return-widget.js` сам добавляет кнопку возврата. Это позволит не переписывать каждый проект вручную.

Поведение:

- если пользователь нажал `×`, виджет скрывается через `sessionStorage`;
- после обновления страницы виджет снова может появиться;
- если нужно скрыть до конца сессии — использовать `sessionStorage`;
- если нужно скрыть навсегда — использовать `localStorage`, но для начала лучше `sessionStorage`.

## Дизайн-система

Нужна своя маленькая дизайн-система для HUB.

### Цвета статусов

- `main` — зелёный / синий акцент: основной рабочий проект;
- `experiment` — жёлтый / янтарный: эксперимент;
- `archive` — серый / графитовый: архив;
- `review` — красный / розовый: требует проверки;
- `featured` — фиолетовый / золотой: избранное.

### Визуальные элементы

- бейдж статуса;
- бейдж категории;
- бейдж версии;
- индикатор наличия скриншота;
- индикатор языка;
- индикатор технологии;
- индикатор `есть архивные версии`;
- индикатор `нужна проверка`;
- кнопка запуска проекта;
- кнопка открытия карточки подробнее;
- кнопка копирования ссылки.

### Кнопки

Кнопки должны быть не просто прямоугольниками, а визуально приятными элементами:

- с заливкой;
- с мягкой тенью;
- с понятным hover-эффектом;
- с иконкой;
- с состояниями `обычная`, `наведена`, `нажата`, `отключена`;
- с цветовой логикой по действию.

Примеры типов кнопок:

- `Открыть проект` — главная яркая кнопка;
- `Подробнее` — вторичная кнопка;
- `Тег` — компактная кнопка-фильтр;
- `Архив` — спокойная серая кнопка;
- `Скопировать ссылку` — техническая кнопка;
- `Показать версии` — кнопка для связанных проектов.

## Что ещё стоит добавить

### 1. Режим подробной карточки

При клике на карточку можно открывать не сразу проект, а подробную панель:

- скриншот;
- описание;
- теги;
- статус;
- технические данные;
- основная версия;
- архивные версии;
- кнопка запуска.

А на самой карточке оставить отдельную кнопку `Открыть`.

### 2. Избранные проекты

Добавить поле:

```js
featured: true
```

И отдельный блок сверху: `Избранное`.

### 3. Уровень готовности

Добавить поле:

```js
maturity: "ready" | "beta" | "draft" | "broken"
```

Это лучше, чем смешивать готовность со статусом архива.

Пример:

- `status: main`, `maturity: ready`;
- `status: experiment`, `maturity: beta`;
- `status: archive`, `maturity: ready`;
- `status: review`, `maturity: broken`.

### 4. Проверка битых ссылок

В локальном инструменте нужна проверка:

- открывается ли проект;
- есть ли `index.html`;
- есть ли скриншот;
- не пустой ли title;
- не повторяется ли id;
- не битая ли ссылка на imageUrl.

### 5. Визуальный режим сравнения версий

Для проектов вроде `3501` и `3502`, `80` и `81` добавить связь:

```js
versionOf: "3502"
```

Тогда HUB сможет показать:

- основная версия;
- старые версии;
- чем отличаются;
- открыть архивную версию.

### 6. Быстрые коллекции

Сверху можно сделать красивые коллекции:

- `Для полиграфии`;
- `Игры`;
- `Для семьи`;
- `Для текста`;
- `Файлы и PDF`;
- `Церковь`;
- `Эксперименты`.

### 7. Мини-панель статистики

Показывать:

- всего проектов;
- основных;
- архивных;
- экспериментов;
- без скриншота;
- требуют проверки.

### 8. Режим плотности

Для дизайнера полезно иметь переключатель:

- большие карточки;
- компактные карточки;
- таблица.

### 9. Темы оформления

Можно добавить темы:

- Steam Dark;
- Print Studio;
- Clean Light;
- Neon Lab.

Сначала достаточно одной хорошей тёмной темы.

## Что может быть лишним

На данном этапе лишним может быть:

- слишком сложная авторизация;
- серверная база данных;
- перенос всех папок;
- удаление старых версий;
- сложные фреймворки;
- попытка сразу сделать идеальную админку;
- слишком много анимаций;
- автоматическая публикация в GitHub без ручного контроля.

## Что может быть нерабочим или рискованным

- Автоматически изменять все старые HTML-файлы без проверки.
- Массово вставлять виджет возврата в сотни файлов без backup.
- Полностью доверять автоматическому определению статуса проекта.
- Делать скриншоты только по GitHub Pages: некоторые проекты могут требовать локального запуска.
- Хранить данные только внутри HTML: потом будет трудно администрировать.
- Использовать слишком тяжёлый интерфейс: HUB должен быть красивым, но не зависать.

## Основные части системы

### 1. `hub-v2.html`

Главная пользовательская витрина.

Должна показывать проекты как визуальные карточки. В ней должны быть:

- скриншоты;
- иконки;
- статусы;
- категории;
- теги;
- поиск;
- фильтры;
- облако тегов;
- кнопка открытия проекта;
- режим просмотра основных проектов, экспериментов и архива.

### 2. `hub-admin.html`

Ручная админка без сервера.

Должна помогать редактировать каталог:

- добавить проект;
- изменить название;
- изменить путь;
- изменить категорию;
- изменить статус;
- добавить описание;
- добавить теги;
- добавить ссылку на скриншот;
- импортировать текущий JSON/JS;
- экспортировать готовый файл данных.

Эта админка не должна сама записывать в GitHub. Она должна экспортировать готовый файл, который можно загрузить или вставить в репозиторий.

### 3. Локальный инструмент на ПК

Это отдельный проект, который запускается на компьютере пользователя.

Возможные технологии:

- Node.js;
- Playwright для автоматических скриншотов;
- простой локальный веб-интерфейс;
- чтение папки репозитория;
- генерация JSON/JS базы.

Что он должен делать:

- сканировать папки и HTML-файлы;
- читать `<title>` из HTML;
- находить существующие `preview.jpg`, `stilist_*.jpg`, `Screenshot.jpg`;
- открывать проекты локально или через GitHub Pages;
- делать скриншоты через браузер;
- создавать каталог проектов;
- выявлять дубликаты;
- предлагать статус `main`, `archive`, `experiment`, `review`;
- экспортировать готовый `projects-hub.js`;
- сохранять скриншоты в папку `assets/hub/screenshots/`.

## Локальный путь работы

1. Пользователь скачивает или клонирует репозиторий на ПК.
2. Запускает локальный инструмент.
3. Инструмент сканирует проект.
4. Инструмент находит HTML-страницы и папки.
5. Инструмент делает скриншоты.
6. Инструмент собирает базу проектов.
7. Пользователь проверяет базу в локальном интерфейсе.
8. Пользователь экспортирует готовый файл `projects-hub.js`.
9. Файл переносится в GitHub.
10. `hub-v2.html` показывает готовую витрину.

## Что делать сначала

Сначала нельзя бросаться в автоматизацию без схемы данных.

Правильный порядок:

1. Утвердить схему данных карточки проекта.
2. Сделать `projects-hub.js`.
3. Сделать `hub-v2.html`.
4. Сделать ручную админку.
5. Сделать локальный инструмент сканирования.
6. Добавить автоматические скриншоты.
7. Связать всё в один рабочий процесс.

## Отношение к старому `shop.html`

Старый `shop.html` не удалять.

Он полезен как визуальный образец и источник данных:

- стиль карточек;
- скриншоты;
- структура старого массива `projects`;
- идея витрины в стиле Steam.

Новый `hub-v2.html` должен взять лучшее из `shop.html`, но стать более системным и управляемым.

## Отношение к текущему `hub.html`

Текущий `hub.html` — это не финал.

Это технический прототип, который проверил идею статусов:

- основные;
- эксперименты;
- архив;
- все.

На его основе можно делать `hub-v2.html`, но визуально и функционально он должен стать намного богаче.

## Автоматические скриншоты

Лучший путь — локальный инструмент на Node.js + Playwright.

Примерная логика:

1. Запустить локальный сервер.
2. Открыть проект в Chromium.
3. Подождать загрузку.
4. Сделать скриншот.
5. Сохранить в `assets/hub/screenshots/{id}.jpg`.
6. Записать путь к скриншоту в базу.

## Что НЕ делать

- Не удалять старые проекты.
- Не переносить папки до завершения базы.
- Не менять `index.html` без отдельного решения.
- Не заменять `shop.html`, пока `hub-v2.html` явно не станет лучше.
- Не полагаться только на ручную память.

## Текущий маршрут

### Этап A — Документация

Создать этот документ и зафиксировать цель.

### Этап B — Данные

Создать `data/projects-hub.js` с расширенной схемой карточек.

### Этап C — Витрина

Создать `hub-v2.html` с красивым интерфейсом, тегами, фильтрами и скриншотами.

### Этап D — Ручная админка

Создать `hub-admin.html` для редактирования и экспорта базы.

### Этап E — Локальный автоматизатор

Создать `tools/hub-builder/`, который будет запускаться на ПК и помогать автоматически собирать базу и скриншоты.

### Этап F — Виджет возврата в HUB

Создать `assets/hub/hub-return-widget.js` и позже безопасно подключать его к проектам.

### Этап G — Финализация

После проверки сделать `hub-v2.html` основной витриной или аккуратно заменить старый `shop.html`.

## Главный итог

Мы строим не просто страницу. Мы строим систему управления коллекцией мини-проектов.

Главная ценность — не потерять старое, но сделать всё видимым, понятным, красивым и управляемым.
